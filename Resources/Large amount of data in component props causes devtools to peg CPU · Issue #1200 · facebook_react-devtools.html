vaughnkoch commented on Oct 27, 2018

Problem:
If a React component has a large amount of data passed in props, even by reference, the CPU gets pegged after the component renders.

Repro

  • Create an array or object structure with a large amount of data, for example 10-20MB.
  • Pass the array variable as the props of a component.
  • Render the component.

This produces the following error:

Uncaught Error: Message length exceeded maximum allowed length.
    at PortImpl.postMessage (VM27 extensions::messaging:121)
    at Port.publicClassPrototype.(anonymous function) [as postMessage] (extensions::utils:138:26)
    at handleMessageFromPage (contentScript.js:24)

The bottom function is this:

    function handleMessageFromPage(evt) {
        evt.source === window && evt.data && "react-devtools-bridge" === evt.data.source && port.postMessage(evt.data.payload);
    }

Here's the current implementation of postMessage which is currently stringifying all the props for the component. Even if they're passed by reference, JSON.stringify still has to process all the data.

  PortImpl.prototype.postMessage = function(msg) {
    if (!$Object.hasOwnProperty(ports, this.portId_))
      throw new Error(kPortClosedError);

    // JSON.stringify doesn't support a root object which is undefined.
    if (msg === undefined)
      msg = null;
    msg = $JSON.stringify(msg);
    if (msg === undefined) {
      // JSON.stringify can fail with unserializable objects. Log an error and
      // drop the message.
      //
      // TODO(kalman/mpcomplete): it would be better to do the same validation
      // here that we do for runtime.sendMessage (and variants), i.e. throw an
      // schema validation Error, but just maintain the old behavior until
      // there's a good reason not to (http://crbug.com/263077).
      console.error('Illegal argument to Port.postMessage');
      return;
    }
    var error = messagingNatives.PostMessage(this.portId_, msg);
    if (error)
      throw new Error(error);
  };

Notes/Thoughts

  • A large amount of data in props passed by ref can be a legitimate use case, for example using Vega for dataviz. Usually this isn't a problem since they're passed by ref.
  • Are there any other ways to serialize arbitrary JS objects instead of JSON.stringify, for example using Apache Arrow?
  • If the props are only used for display to the user in the inspect or profiler ui, would it make sense to just send the first 10-20 elements in an array and tell the user that the data was clipped? Not sure how you'd handle this with arbitrarily large objects though.

edmorley on Nov 9, 2018

Hi :-)

This issue sounds like what is preventing us from using react-devtools with mozilla/treeherder.
(Happy to file as a separate issue if preferred?)

STR:

  1. Clone https://github.com/mozilla/treeherder (tested using 3ad320d)
  2. yarn
  3. yarn start
  4. In the new browser tab opened by webpack-dev-server (which should end up redirecting to http://localhost:5000/#/jobs?repo=mozilla-inbound), open devtools and switch to the React tab
  5. Switch to the "Profiler" sub-section within the React tab
  6. Click "Start profiling" and then 2-3 seconds later, click "Stop profiling"

Expected:

  • Browser behaviour / responsiveness is roughly the same as when not using react-devtools
  • No console errors from react-devtools
  • A recording is created and the results of which can be viewed

Actual:

  • "Connecting to React‚Ä¶" takes 30+ seconds (and sometimes doesn't complete at all)
  • Browser extremely unresponsive (eg scroll action takes 10-20 seconds to respond)
  • High Chrome CPU/RAM usage
  • No recording shown after pressing "Stop profiling"
  • Console errors:
Uncaught Error: Message length exceeded maximum allowed length.
    at PortImpl.postMessage (VM1326 extensions::messaging:121)
    at Port.publicClassPrototype.(:5000/anonymous function) [as postMessage] (extensions::utils:138:26)
    at handleMessageFromPage (contentScript.js:24)

Versions:

  • React 16.6.1
  • Chrome beta 71.0.3578.44
  • Chrome React Developer Tools 3.4.2 (from store)
    (Also reproduces using Firefox Nightly + Firefox React Developer tools)

The app needs some performance work (it's a legacy app that has just undergone a conversion from AngularJS 1.x to React; we're wanting to explore things like react-window next) which would presumably also help avoid this issue -- but it's hard to do so when unable to use react-devtools to start with :-)

cc @camd

bvaughn on Dec 13, 2018

Contributor

Potentially. I had two ideas that might be worth trying, but I haven't tried either (or thought about either in much depth).

  1. We could serialize/send less data for each React commit by making props and state objects lazy (like we already do in some cases for nested values) and‚Äì rather than send them automatically, wait until the extension requests them.
  2. We could send data less often by throttling the updates (rather than serializing for each commit). This would probably be fine for the Elements view, since eventual consistency is probably fine for that kind of UI. This might have implications for the Profiler that would require more thought.
